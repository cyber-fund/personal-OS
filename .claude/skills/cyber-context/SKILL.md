# Cyber Context Skill

Show active projects dashboard with recent communication signals. Helps the user understand focus, priorities, and what's happening across projects.

**Usage:** `/cyber-context`

---

## Workflow

### Phase 1: LOAD PROJECTS

Read all project status files from `./vault/private/projects/*/status.md`. From each, capture:
- The `## Summary` section (the rolling project summary) — used for the index rollup
- Current state, In progress, Next

For each project also read the `README.md` to get:
- Project name and one-liner
- Success criteria
- Signals section (if exists): `people` and `keywords` lists for matching against inbox

Sort projects by `status.md` file modification date (most recent first).

### Phase 2: LOAD RECENT INBOX

Read inbox files from `./vault/content/inbox/`. Load the last 3 days of inbox files (e.g. `2026-05-23.md`, `2026-05-22.md`, `2026-05-21.md`).

These contain raw deltas from Telegram, Email, Meetings (Granola), and Twitter — organized by source with message text.

### Phase 3: MATCH SIGNALS

For each project that has a `## Signals` section in its README.md:
- Extract `people:` list and `keywords:` list
- Scan inbox entries for mentions of those people or keywords
- Collect matching entries as "relevant signals" for that project

Also identify inbox entries that don't match any project — these are "unmatched signals" that might indicate new work or priorities the user hasn't tracked yet.

### Phase 4: REGENERATE PROJECTS INDEX

Rebuild `./vault/private/projects/INDEX.md` from the `## Summary` section of every project's `status.md`. This is a derived rollup — overwrite it in full each run. Sort by most-recent activity (status.md mtime).

```
# Projects Index

Rolling summary of all projects. Regenerated by /cyber-context from each
project's status.md `## Summary`. Source of truth = the per-project status.md.

**Last updated:** YYYY-MM-DD

## <Project name> — <status> (updated YYYY-MM-DD)
<Summary text from that project's status.md>
→ [README](./<slug>/README.md) · [status](./<slug>/status.md)

## <next project>
...
```

If a project has no `## Summary` yet, use its Current state as a fallback and note `(no summary)`.

### Phase 5: OUTPUT DASHBOARD

Output a structured dashboard:

```
# Context Dashboard — YYYY-MM-DD

## Active Projects (by recent activity)

### 1. Project Name
**Status:** [from status.md current state]
**Last activity:** [date from file mtime]
**In progress:** [from status.md]
**Next:** [from status.md]
**Recent signals:**
- [source] Person said "..." (date)
- [source] Email from X about Y (date)

### 2. Project Name
...

## Unmatched Signals

Recent messages/emails/meetings that don't map to any project:
- [telegram] @username: "message preview..." (date)
- [email] From: name — Subject (date)
- [meeting] "Meeting title" with attendees (date)

## Stale Projects

Projects with no activity or signals in the last 14 days:
- Project Name (last activity: date)

## Suggested Actions

Based on the signals and project states:
- "Consider linking @person to project X — they discussed related topics"
- "Project Y has been stale for 3 weeks — archive or re-prioritize?"
- "Unmatched signal about Z might warrant a new project"
```

### Phase 6: OFFER ACTIONS

After showing the dashboard, ask the user if they want to:
1. Update any project's status
2. Add signals (people/keywords) to a project
3. Re-prioritize (move projects between Now/Next/Someday in GTD)
4. Create a new project from unmatched signals

---

## Notes

- The Signals section in project README.md is optional. Projects without it won't get signal matching but will still appear in the dashboard based on file activity.
- The skill only writes the derived `projects/INDEX.md` rollup (Phase 4). It does NOT modify any project files unless the user asks for an action in Phase 6.
- Keep the dashboard concise — truncate message previews to ~100 chars.
- If no inbox files exist yet, skip the signals section and note that the user should run the collect script first.
